The E3D Ecosystem —
A Navigation-to-Execution Stack for On-Chain Markets

E3D is a starship. It is assembled from the bottom up and it flies from the bottom up. Fuel is forged on-chain, ignited into intelligence, focused into guidance, and spent at the command tip where decisions are made. Spent fuel is not wasted — the exhaust is recaptured as training signal, and the vehicle climbs a little smarter on every burn.

Overview

Four Stages, One Vehicle

E3D turns raw on-chain evidence into autonomous, accountable action — and then learns from the result. It is not a single application but a layered stack of four independent repositories, each a stage of the same vehicle, bolted together not by shared code but by shared contracts.

The animating thesis is simple and unfashionable: on-chain story signals fire before price moves. Patterns in wallet behavior, token flows, and market microstructure precede the candle. E3D is built to be a telescope pointed at the pre-pump window — not a rear-view mirror reacting to moves already in progress.

🚀 Engines

E3DToken — The Economic Substrate

Every rocket needs propellant with a known chemistry. E3D's propellant is a single canonical asset with an unambiguous definition across chains. The rules are simple: one origin, one supply, one source of truth. Every other layer indexes it, narrates it, or trades it — but none of them define it.

E3DToken.sol — the canonical ERC-20

A clean OpenZeppelin ERC-20 (E3D) minted on Ethereum — the source of truth for supply and liquidity. Two extensions do real work in the agent economy: ERC20Burnable powers deflation-on-usage (fees are burned, so genuine activity makes the token scarcer), and ERC20Permit lets an agent authorize a spend with a signature instead of a separate transaction — the gasless primitive that machine-to-machine x402 payments ride on.

E3DNFTManager.sol — the agent identity & payment layer

An upgradeable (proxy) ERC-721 that is now the on-chain home for autonomous agents — far more than the NFT marketplace it began as. Each agent mints a verifiable identity (mintAgentIdentity) carrying its controlling wallet and a registration URI; the activation fee is burned in E3D, so identity costs scarcity.

Reputation, validation level, and activity are recorded on the token, written by a low-privilege scorer role (the Maps scoring service) kept separate from the admin key. getAgentTier exposes an access tier that off-chain x402 metering reads to price each navigation call — you pay more for proven agents, not flat-rate access. Identity tokens are non-transferable except through a deliberate handoffAgent (you can sell a proven agent, but it can't be casually listed), and an agent's tokenURI resolves to maps.e3d.ai/agents/<id> — living metadata that serves its current reputation.

The result is the payment model for agentic navigation: E3D is the unit, x402 the rail, the NFT is identity + reputation, and burn returns value to holders — one token that meters what an agent consumes, prices it by what the agent has earned, and rewards what it honestly contributes, with no human in the loop.

Wormhole bridge — the cross-chain rails

Rather than fragmenting supply by minting a second independent token, E3D uses Wormhole Wrapped Token Transfers. The canonical asset stays locked on Ethereum; a 1:1-backed wrapped SPL representation is minted on Solana after verification. The flow is deliberate and auditable: attest → transfer → redeem → verify balances on both chains. Ethereum is canonical. Solana receives the wrapped version, not a new token.

This layer is the substrate the rest of the stack indexes, narrates, predicts over, and ultimately trades. Combustion happens here; everything above is what we do with the energy.

🛰 Core stage

spacepacket / E3D.ai — The Data and Intelligence Hub

This is the heavy stage — the largest tank in the vehicle and the gravitational centre of the whole ecosystem. It does three jobs: ingest, understand, and serve. Raw rows from the chain arrive here and leave as stories, theses, and wallet intelligence that every upper layer can trust.

Ingestion — buildDB

A wide battery of builders pulls the chains into a queryable substrate: Ethereum, BSC, and Solana transactions and swaps; price histories; pool liquidity; address names and symbols; and rolling timeseries that give every later layer a sense of motion rather than a single snapshot. This is not a log. It is a living map of capital.

Intelligence

Raw rows are not insight. The intelligence layer distills activity into the higher-order objects the rest of the stack consumes:

StoriesTyped on-chain narratives
ThesesHigh-conviction views
Wallet intelCounterparties + flows
Market intelMicrostructure signals
OutcomesWhat actually happened

Stories are the nervous system. They are typed narratives detected in on-chain behavior — staging, clustering, accumulation, smart-money, stealth accumulation, breakouts, and more. These are the pre-pump signals the entire thesis rests on. A token with an active story is a token where something is happening. Everything else is noise.

Serving — API, MCP, and the visualization

The core stage exposes its intelligence through an HTTP API and a Model Context Protocol (MCP) surface, so both humans and agents can read the same ground truth. The public /api/maps/... routes live here, in the main E3D repository — the upper stages produce objects that this surface later serves. On top of it all sits the original E3D promise: a 3D visualization that turns Ethereum data into a navigable landscape, where users can explore wallets, tokens, and flows as terrain.

The stage that converts fuel into usable thrust. Everything above it is a consumer of its intelligence. Everything it learns flows back down into it.

🧭 Guidance

e3d-maps — Navigation Intelligence

If the core stage tells you what is happening, Maps tells you where things are going next. It is the guidance computer: it reads the evidence and forecasts the routes capital is likely to take, where congestion is forming, and which paths are becoming hazardous for downstream agents.

Maps is, by design, a producer of validated machine-readable objects — never vibes, never prose predictions. Every output is strictly schema-checked before any database write. Confidence values are bounded. Required fields are enforced. Unknown signal types are rejected. The schema is the contract.

The agent swarm

A constellation of focused producer agents, each with a narrow mandate, orchestrated over a Qwen runtime with per-request LoRA adapter selection:

CapitalMigration + conviction
CongestionFormation detection
RoutesEmergence + closure + hazard
DestinationProbability prediction
LiquidityForecast
NarrativeAcceleration
WatchDraft + outcome scoring
ConfidenceCalibration scoring

The objects it emits

The shared schema layer is the heart of the repository. Every output validates against it before any write happens:

NavigationSignal · RoutePrediction · TrafficState · PredictionOutcome · SignalUtilityScore · FlowGraph · CrossChainActivityState · WatchDraft · WatchPrediction · MapsNewsBrief

Scheduled jobs and the closed loop

Scheduled jobs assemble traffic state and flow graphs, generate navigation signals, backtest predictions, score pending and settled predictions, compute signal utility scores, and export training examples. This last step is what makes Maps a learning system rather than a static oracle: settled predictions become labeled training data that feeds the next generation of the guidance computer.

Maps ships its own frontend to maps.e3d.ai, served as static files by nginx that proxies API calls back to the main E3D endpoint. Same-origin, zero coupling between the static layer and the API runtime.

🎯 Command tip

e3d-agent-trading-floor — Where Intent Becomes Consequence

The nose of the vehicle — the only stage permitted to act. It is a multi-agent, AI-assisted portfolio system that runs a continuous cycle of discovery, evaluation, risk validation, and execution. It is paper-mode by default. Live execution requires an explicit, deliberate configuration change. Within paper mode, all portfolio state mutations are real — the P&L tracking, position sizing, cooldowns, and rotation logic all function as they would in live trading. The only thing missing is the actual order submission.

Five specialists, one deterministic spine

No agent works alone, and no agent can unilaterally move capital. Five LLM agents collaborate inside a pipeline whose backbone is plain, auditable code:

Agent Mandate
Scout Discovers 0–3 buy candidates from a story-filtered universe
Harvest Manages held positions — hold, monitor, trim, or exit
Risk Enforces hard limits and quant gates — the kill switch in code
Executor Records the paper trade ticket or submits the live order
Manager Grades the cycle after the fact, flags issues, writes the report

Between the agents sit the parts that actually hold the keys: a perception layer that builds a ranked cognitive state from a handful of targeted API calls, hard sell checks (stop-loss, fraud breach, target hits), and a deterministic portfolio engine for ranking, rotation, and allocation.

Story-first, evidence-chained

The token universe shown to Scout is not a list of high-volume movers — it is dynamically assembled from active story coverage and sorted by freshness of on-chain signal. Tokens with no story are excluded regardless of momentum. Every candidate must carry an evidence chain: evidence[], why_now, risks[], conviction_score, an entry zone, an invalidation price, and targets. Undocumented decisions are invalid by construction.

The core law of the command tip

AI suggests. Code decides. Language models produce structured JSON proposals. Deterministic pipeline code validates every field, enforces every hard limit, and chooses whether anything happens at all. A hallucination, a prompt injection, or a degraded model can waste a cycle. It cannot move capital. This separation is not a safeguard bolted on after the fact — it is the foundational architecture decision from which everything else follows.

External boosters

Strap-on quant feeds enrich each cycle: DexScreener order flow, CoinGecko, Fear & Greed, and Binance funding. Every cycle is UUID-stamped and logged append-only to pipeline.jsonl and optionally to ClickHouse for training data retention. A live dashboard renders the floor in motion.

Agentic loops

Three Loops, Three Tempos — The Unit of Computation

Strip away the metaphor and E3D is not really four programs. It is three agentic loops running at three different tempos, chained end to end. Each loop is the same primitive — perceive → decide → act → score → learn — and one loop's output is the next one's input. The loop, not the model, is the unit of computation.

The core loop — spacepacket, continuous

Ingest on-chain activity → detect stories and theses → generate candidate actions → evaluate what actually happened → recalibrate → ingest again. The agent runtime and outcome evaluators turn a firehose of transactions into scored intelligence, and never stop.

The navigation loop — e3d-maps, scheduled

Pull evidence → the producer swarm emits signals and route predictions → consumers act → dual-witness scoring settles each prediction → calibration and utility scores update → quality-gated examples export → the adapter fine-tunes → sharper predictions next cycle. This is the loop that learns to be right.

The trading loop — e3d-agent-trading-floor, every ~5 minutes

Build the cognitive state → Scout and Harvest propose → Risk validates → the portfolio engine sizes → Executor records → Manager grades the cycle → log the training event → run again. A deterministic spine with five agents riding it.

Loops with skin in the game. What separates these from the demos crowding everyone's feed is that they are economically autonomous. Every agent can carry an on-chain identity (an E3DNFTManager NFT), earn a reputation scored by the loop's own outcomes, pay for services it consumes from other loops (x402, settled in E3D, priced by its tier), and be paid for the honest signal it contributes. Perceive, decide, act, score, learn — and settle up.

Nested, not isolated. The loops chain: the core's stories feed Maps' predictions; Maps' predictions feed Trade's decisions; the settled outcomes from Trade and Maps flow back down into the core and the shared adapters. Three tempos, one closed system — which is exactly what the exhaust is.

🔥 The exhaust

Outcomes as Training Fuel — The Closed Loop

A starship that threw its propellant away would be a poor design. E3D recaptures it. Predictions and trades both settle — and settlement is the most valuable data the system produces. Every outcome flows back down the stack as training signal, so the vehicle does not just climb; it climbs smarter on every burn.

The compounding loop

→E3D stories + theses + wallet data + market data (the sensors)
→Maps agents generate NavigationSignals with stated confidence
→Consumer agents act — trading, treasury, research
→Dual-witness outcome scoring: did it realize? (exogenous vs. induced)
→Calibration curves updated; signal utility scores computed
→Quality-gated training export: disputed labels excluded
→Qwen + LoRA adapter fine-tuned on honest examples only
→Sharper signals, better-calibrated confidence, smarter pipeline
↺Repeat — each revolution produces a slightly smarter navigator

The system is designed to be wrong early and right later — to fail with precision, learn from the failure, and accumulate an advantage that compounds with every cycle. The training data asset at the centre of that loop cannot be scraped, synthesised, or purchased. It can only be earned by running the system honestly and scoring every prediction ruthlessly.

The launch infrastructure

How the Stages Connect — Contracts, Not Imports

The stages are bolted together by contracts, not imports. This is a hard architectural law, not a stylistic preference. There are no direct runtime imports between any two repositories. Shared databases and API contracts are the integration boundary. Replace a stage and the contract holds. Couple two stages directly and you have welded the fuel tank to the nose cone — exactly what this architecture refuses to do.

ClickHouseShared write-side store for predictions, outcomes, training data. Append-only.
API + MCPThe lingua franca. Upper stages read ground truth here. Public /api/maps/... routes live in the core.
Qwen + LoRAShared inference substrate. Per-request adapter path. Specialized agents on a common serving layer.

The payoff: each stage can be built, tested, deployed, and reasoned about in isolation. The contract is the only surface that matters.

End to end

One Full Burn — A Signal From Engines to Tip and Back

To see the whole vehicle fire in sequence, follow a single signal from the engines to the command tip and back into the exhaust:

Full cycle

1Ignition. Capital moves on-chain — an E3D token, an NFT, a swap, a bridge transfer.
2Combustion. spacepacket indexes it and detects a story — say, stealth accumulation in the pre-pump window.
3Guidance. e3d-maps reads that story and emits a RoutePrediction and a NavigationSignal: capital is likely to flow toward a given destination, and the route is clear.
4Command. Scout, shown a story-filtered universe and the Maps picture, proposes an evidence-chained candidate. Risk validates against hard limits. The portfolio engine sizes it. Executor records the ticket.
5Exhaust. The prediction and the trade both settle. Maps scores the prediction; Manager grades the cycle. Both outcomes export as training data and flow back down.
6Next burn. Sharper stories, better-calibrated predictions, more disciplined agents.

Design principles

The Laws That Hold the Vehicle Together

→
AI suggests, code decides. Models propose; deterministic code disposes. The LLM never writes directly to the portfolio.
→
Story-first, not price-first. Aim at the pre-pump window, never chase the candle. Tokens with no story coverage are excluded regardless of momentum.
→
Evidence chains, not guesses. Every decision carries its receipts. Undocumented candidates are invalid by construction.
→
Validate before you write. Strict schemas guard every database boundary. A signal that fails validation is never written.
→
Contracts, not imports. Stages meet in shared data and APIs — never in each other's runtime. The integration boundary is explicit and narrow.
→
Fail safely. Paper mode by default. Live execution is an explicit, deliberate act. Within paper mode, all logic is live — only the order submission is withheld.
→
Close the loop. Outcomes are fuel. Recapture the exhaust. The system is designed to be wrong early and right later.
